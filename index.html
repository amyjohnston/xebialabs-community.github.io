Welcome to XebiaLabs community plugins

This page describes how you can contribute to XebiaLabs community plugins, where continuous integration is located, and which steps to follow when releasing your community plugin.

Join the xebialabs-community organization

Create a GitHub account and request to join the organization by sending an email to jdewinne@xebialabs.com.

Fork a repository or generate a pull request

If you want to contribute to an existing repository, simply fork the repository, make your changes, and send a pull request. If you need help, try contacting the person who last committed to that repository.

License to use

IMPLIED, INCLUDING BUT NOT LIMITED TO THE IMPLIED WARRANTIES OF MERCHANTABILITY AND/OR FITNESS

You can easily include this information by using the Gradle license plugin.

buildscript {
  repositories {
    jcenter()
  }
  dependencies {
    classpath "nl.javadude.gradle.plugins:license-gradle-plugin:0.11.0"
  }
}

apply plugin: "com.github.hierynomus.license"

Continuous integration

Travis CI

The XebiaLabs community plugins use Travis CI as a continuous integration tool. Go to the Travis CI website and log in with your GitHub account. After you log in, you will see the community plugins that are being built. You can find the configuration for Travis CI in the .travis.yml file in each repository. For more information, refer to the Travis CI documentation.

Also, please configure HipChat notifications (TravisCI room) and GitHub releases.

BuildHive

Alternatively, you can use BuildHive (Jenkins in the Cloud for open source); however, it may not be possible to integrate with HipChat or send releases to GitHub.

Use a Gradle wrapper

You can find an example of a Gradle wrapper in the xld-cloud-foundry-plugin repository. Using a Gradle wrapper ensures that a consistent Gradle version is used, including on Travis CI. You can copy the sample wrapper from the xld-cloud-foundry-plugin repository or create your own using gradle wrapper.

Release your plugin

To release your plugin:

  • Update the build.gradle file to contain the new version number.
  • Update the .travis.yml file to contain the new version number.
  • Create a new tag; for example, git tag -a v4.5.2 -m 'Version 4.5.2' 373159.
  • Push the tag to GitHub: git push --tags.

Rules to follow

  • Each repository must include the license
  • If your repository is a plugin, its name should end with plugin
  • If your repository is a plugin for XL Deploy, its name should start with xld
  • If your repository is a plugin for XL Release, its name should start with xlr
